## 固件升级（Bootloader）交互协议说明

### 1. 升级流程概述

升级过程采用 **“请求-响应”** 机制。上位机作为 Master 驱动状态机，下位机作为 Slave 响应指令。

### 2. 交互详细步骤

| 阶段 | 步骤 (上位机动作) | 下位机预期响应 (ACK) | 说明 |
| --- | --- | --- | --- |
| **复位** | 发送 `PROTOCOL_CMD_RESET` | 响应相同命令码 | 强制系统复位或进入 Boot 模式。 |
| **解锁** | 发送 `PROTOCOL_CMD_STARTBL` | 返回 `IronBoot::BlUnlock` (固定码) | 只有解锁成功才允许后续擦除。 |
| **擦除** | 发送 `PROTOCOL_CMD_EARSE` | 响应相同命令码 | 下位机执行 App 区 Flash 擦除，耗时较长。 |
| **预备** | 发送 `PROTOCOL_CMD_BIN` | 响应相同命令码 | 告知下位机即将开始传数据，初始化 Seq。 |
| **传输** | 发送 `PROTOCOL_CMD_DATA` (多帧) | 响应 `Seq` 与 `CRC` 校验 | 分包发送，每包由下位机确认后再发下一包。 |
| **校验** | 发送 `PROTOCOL_CMD_CHECKBIN` | 返回下位机自算 CRC32 | 下位机对整个已写入区进行完整性校验。 |
| **结束** | 发送 `PROTOCOL_CMD_SUCC` | 无或自定义 | 升级完成，下位机准备跳转。 |

---

### 3. 数据帧格式约定

#### 3.1 基础命令帧 (CmdFrame)

用于握手和控制。下位机应答时，数据段需回传对应的状态或命令码。

* **Cmd**: 包含 `RESET`, `STARTBL`, `EARSE`, `BIN`, `CHECKBIN` 等。

#### 3.2 数据传输帧 (DataFrame)

* **Seq**: 包序号（从 0 开始自增）。
* **Len**: 本帧有效负载长度（固定长度，最后一帧除外）。
* **Data**: 固件原始数据（不足一帧长度时，上位机已填充 0x00，下位机应按 **Len** 写入）。
* **CRC**: 本帧数据的校验和（非整片校验）。

---

### 4. 下位机实现要点 (给开发者的建议)

1. **分包写入**：由于上位机采用“停-等”协议，下位机必须在 Flash 写入完成并校验当前帧 CRC 无误后，再回复 ACK。
2. **最后一帧处理**：固件总长不一定是 128/256 的倍数，请严格根据数据帧里的 `Len` 字段进行局部写入，避免多写多余的 `0x00` 导致整片 CRC 错误。
3. **超时控制**：在 `EARSE`（擦除）阶段，下位机可能无法响应通讯，请告知上位机预留足够的擦除时间（当前上位机超时设为 10 秒）。
4. **完整性校验**：收到 `PROTOCOL_CMD_CHECKBIN` 后，请对接收到的整个 App 区域进行标准 CRC32 计算，并将结果回传，上位机会进行二次比对。
5. **重发机制**：如果收到错误的 CRC 或 Seq，请回复错误码或不回复，触发上位机自动重发（上位机单步重试上限为 5 次）。

---

### 5. 错误码定义

* **超时**: 10s 无应答视为失败。
* **重发次数溢出**: 连续 5 次数据包确认失败。
* **校验失败**: 整片 CRC 结果与上位机计算不符。
